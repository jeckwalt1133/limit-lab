# Agentic拓扑优化 — 2026研究共识

## 核心发现: 更多Agent ≠ 更好

2026年多个研究组得出同一结论: 盲目增加Agent带来的是边际递减甚至灾难性失败。

## 三个关键效应

| 效应 | 描述 |
|------|------|
| 工具-协调权衡 | 多工具任务在多Agent下反而性能更差 (Google/MIT 2026) |
| 能力饱和 | 一旦单Agent基线超过阈值，加Agent收益递减 |
| 拓扑依赖误差放大 | 集中式编排减少误差放大；去中心化放大误差 |

## 最优Agent数量

| 规模 | 架构 | Agent数 | 适用 |
|------|------|---------|------|
| 微型 | 集中式单体 | <10 | 我们的当前规模 |
| 小型 | 集中式+消息总线 | 10-50 | 成长后 |
| 中型 | 分布式分片 | 50-100 | 成熟期 |
| 大型 | 混合层级 | 100+ | 未来 |

实际生产数据: 平均部署=23.7个Agent。78%的团队在超过20个Agent时遭遇系统性失败。

## 分解陷阱 (#1失败模式)

❌ 按角色分解: "一个写代码、一个写测试、一个审查" → 协调开销吞噬工作产出
✅ 按上下文分解: 只在实际可隔离上下文时才拆分

Anthropic发现: "子Agent在协调上花的token比实际工作还多"

## 自动化拓扑选择 (2026最新)

- ARG-Designer (AAAI 2026): 用自回归图生成自动决定Agent数+角色+通信链路
- RGAO/Code-Agent: 按5维复杂度向量选4种拓扑之一

## 对我们的意义

**我们当前Alpha+Beta+Merge = 3 Agent，处于最优区间。**

关键决策:
1. 不要盲目加分支 → 先优化现有分支的协调
2. 按上下文拆分(已做对) → 不按角色拆分
3. 未来扩展用自动化拓扑 → 让系统自己决定需要几个分支
4. 每新增一个Agent必须有明确证据 → 上下文污染、并行化需求、或工具过载
