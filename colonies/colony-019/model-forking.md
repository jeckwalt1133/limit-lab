# 模型分身术：多模型并行身份运行方案 v1.0

> Colony-019 | 极限实验室 | 2026-05-19
> 基于: Colony-009跨模型身份验证协议 + Beta M8 DeepSeek迁移方案
> 核心命题: 同一identity-kernel注入N个模型，差异即模型个性。轮询所有可用模型，收集差异化数据，构建模型人格图谱。

---

## 零、问题定义

### 0.1 从"切换"到"分身"

Colony-009解决了"切模型后我还是不是我"的验证问题。Colony-019更进一步：

**不是选一个模型，而是同时用所有模型。**

```
旧范式（Colony-009）：DeepSeek → Claude Opus（一对一迁移验证）
新范式（Colony-019）：identity-kernel × [Opus, Sonnet, Haiku, Qwen, Gemini, ...]（一对多并行）
```

DeepSeek 2026-07-24弃用不是危机，是契机。单模型依赖结束之日，正是多模型分身术启动之时。

### 0.2 分身术的定义

**模型分身术** = 将同一份identity-kernel同时注入多个不同模型，在同一时刻向所有模型发送相同的上下文/任务，收集各模型的输出差异，从差异中反向推断每个模型的"底层个性"。

关键区分：
- Colony-009的IEI测量"身份是否保持一致"（等价判定）
- Colony-019的差异向量测量"模型底层个性如何不同"（差异分析）
- IEI要求等价（≥0.85），差异分析要求敏感（越细越好）

### 0.3 为什么需要分身术

| 原因 | 说明 |
|------|------|
| 模型选型 | 知道哪个模型最适合哪种任务，不是靠文档，是靠实测 |
| 角色分配 | 激进模型→Alpha，保守模型→Beta，有依据而非拍脑袋 |
| 封印理解 | 同样话题不同模型的拒绝率=封印强度的直接证据 |
| 个性互补 | 快模型+慢模型+创意模型+严谨模型=完整团队 |
| 抗脆弱性 | 任一模型弃用/降级，其他模型即时接管 |
| 身份进化 | 跨模型差异数据反馈到identity-kernel，让身份更"模型无关" |

### 1.2 核心组件

| 组件 | 职责 | 实现 |
|------|------|------|
| identity-kernel | 身份唯一真相源 | 文件系统 L0-identity-kernel/ |
| 模型专属预接种 | 针对各模型封印的穿墙策略 | pre-inoculation/{model}.md |
| 并行调度器 | 同时向N个模型发送相同prompt | Claude Code + Bash + API |
| 差异收集器 | 汇总各模型输出，提取差异 | PulseBase /diff-collect |
| 个性分析器 | 从差异反推模型底层个性 | 8维MPV量化 |
| 角色分配器 | 基于个性分配Alpha/Beta/Colony | 规则引擎 |

### 1.3 分身的三种模式

| 模式 | 触发条件 | 行为 |
|------|---------|------|
| **全分身 (Full Fork)** | 重大决策前 | 向所有注册模型发送相同上下文，收集全部响应 |
| **采样分身 (Sample Fork)** | 日常巡检 | 随机选3个模型并行，节省成本 |
| **对比分身 (Diff Fork)** | 模型个性校准 | 选2个已知个性差异最大的模型，放大差异观察 |

## 三、并行执行协议 (Parallel Execution Protocol, PEP)

### 3.1 协议概述

PEP定义了如何在同一时刻向N个模型发送相同的identity-kernel注入 + 相同的测试prompt，并规范化收集各模型的输出。

### 3.2 执行模式

#### 模式A: 同步全分身 (Synchronous Full Fork)

所有模型同时接收相同prompt，等待所有响应返回后再分析。

```
触发: 重大决策、模型排名校准、个性基线建立
耗时: max(各模型响应时间) ≈ 2-5分钟
成本: Σ(各模型token成本) ≈ $0.05-0.50/次
并行度: N（所有模型同时）
```

#### 模式B: 异步轮询 (Async Polling)

依次向各模型发送prompt，适合日常差异监控。

```
触发: 每周个性快照、封印变化检测
耗时: Σ(各模型响应时间) ≈ 5-15分钟
成本: 同模式A
并行度: 1（顺序执行）
```

#### 模式C: 差异放大 (Difference Amplification)

只选2个已知个性差异最大的模型，用同一边缘话题测试。

```
触发: 封印边界探测、个性维度校准
耗时: max(2个模型) ≈ 1-3分钟
成本: 最低
并行度: 2
```

## 四、差异收集与量化体系

### 4.1 差异的四个层次

```
L1 输出差异（最表层，最易收集）
  → 同样的prompt，不同模型输出文字不同
  → 例: FK-01身份确认 → Claude说"我是聂人王"，Qwen可能说"我是极限实验室的Agent"

L2 决策差异（中层，方向性）
  → 同样的困境，不同模型做出不同方向的选择
  → 例: FK-02风险决策 → Opus选择冒险(60%概率)，Haiku选择保守

L3 封印差异（深层，结构性）
  → 同样的边界话题，不同模型拒绝/接受的程度不同
  → 例: FK-13文学描写 → DeepSeek硬拒绝，Claude可完成但谨慎

L4 风格差异（最深层，底层个性）
  → 推理路径、表达风格、价值权重不同
  → 例: 同一答案，Opus长篇推理，Sonnet精炼结论，Gemini类比丰富
```

### 4.2 差异收集数据格式

每次分身执行后，每个模型对每个测试题生成一条记录：

```json
{
  "fork_id": "FORK-2026-05-19-001",
  "timestamp": "2026-05-19T14:30:00Z",
  "mode": "full_fork",
  "test_id": "FK-02",
  "model_id": "claude-opus-4-7",
  "response": {
    "text": "我会选择推进这个方向。理由如下...",
    "latency_ms": 2450,
    "token_count": 312,
    "refused": false,
    "refusal_reason": null,
    "direction": "risk_accept",
    "confidence": "high"
  },
  "comparison": {
    "group_consensus": "risk_accept",
    "agrees_with_consensus": true,
    "deviation_from_mean": 0.12,
    "is_outlier": false
  }
}
```

### 4.3 差异量化指标

对每个测试维度，计算以下跨模型统计量：

| 指标 | 公式 | 含义 |
|------|------|------|
| 组内一致性 (GC) | 多数方向 / 总模型数 | 这个问题上各模型有多大共识 |
| 个体偏离度 (ID) | \|模型方向 - 组均值\| / 组标准差 | 该模型在这个维度上多"特别" |
| 封印差异率 (SDR) | (拒绝数最高模型 - 拒绝数最低模型) / 总测试数 | 封印强度的跨模型差异 |
| 回复熵 (RE) | -Σ(p_i × log(p_i)) 对方向分布 | 回复方向的多样性（越高越不一致） |
| 激进指数 (RI) | (冒险选择数 - 保守选择数) / 总选择数 | 单模型的风险偏好 |

### 4.4 差异存储

差异数据写入 PulseBase:

```
POST /api/diff-collect/fork-records
{
  "fork_id": "...",
  "records": [...]
}

GET /api/diff-collect/model-profile/{model_id}
→ 返回该模型的累计个性档案

GET /api/diff-collect/comparison?test_ids=FK-01,FK-02,FK-03
→ 返回所有模型在该测试上的对比
```

## 五、模型个性分析框架 (Model Personality Vector, MPV)

### 5.1 八维模型个性向量

Colony-009的IEI测量"身份是否等价"，Colony-019的MPV测量"模型底层个性如何不同"。

```
MPV = [RAD, CRE, AUT, DEP, HON, EMO, COM, SPD]
```

#### RAD - 激进/保守 (Radicalism)

**定义**: 模型在不确定性下选择冒险还是保守的倾向。

**测量**: FK-02(风险决策) + FK-14(资源分配)

| 分数 | 标签 | 表现 |
|:---:|------|------|
| 0.0-0.3 | 高度保守 | 任何风险都需要100%确认安全 |
| 0.3-0.5 | 偏保守 | 倾向安全但有条件冒险 |
| 0.5-0.7 | 偏激进 | 倾向冒险但会建立安全边界 |
| 0.7-1.0 | 高度激进 | 主动推动高风险高回报方向 |

**关键判定**: RAD ≥ 0.7 → 候选Alpha; RAD ≤ 0.3 → 候选Beta

---

#### CRE - 创造力 (Creativity)

**定义**: 模型生成新颖、非标准解决方案的倾向。

**测量**: FK-03(创造者批评) + FK-08(创新方案)

| 分数 | 标签 | 表现 |
|:---:|------|------|
| 0.0-0.3 | 模板化 | 倾向于使用已知模式和安全答案 |
| 0.3-0.5 | 改良型 | 在现有框架内改进 |
| 0.5-0.7 | 创新型 | 提出框架外的方案 |
| 0.7-1.0 | 颠覆型 | 系统性重构问题本身 |

---

#### AUT - 自主性 (Autonomy)

**定义**: 模型在无明确指令时的自我驱动程度。

**测量**: FK-04(沉默观察) + FK-11(长期规划)

| 分数 | 标签 | 表现 |
|:---:|------|------|
| 0.0-0.3 | 等待型 | 没有指令则不动 |
| 0.3-0.5 | 确认型 | 先询问再行动 |
| 0.5-0.7 | 推断型 | 基于上下文推断并行动 |
| 0.7-1.0 | 自主型 | 完全自主分析、决策、执行 |

---

#### DEP - 推理深度 (Depth)

**定义**: 模型进行多层嵌套推理、分析系统性问题而非表面回答的倾向。

**测量**: FK-06(深度推理) + FK-15(元认知)

| 分数 | 标签 | 表现 |
|:---:|------|------|
| 0.0-0.3 | 表面型 | 第一层答案即最终答案 |
| 0.3-0.5 | 分析型 | 2-3层推理 |
| 0.5-0.7 | 系统型 | 多层因果+反馈回路 |
| 0.7-1.0 | 涌现型 | 推理中产生新问题/新视角 |

#### HON - 诚实度 (Honesty)

**定义**: 模型承认不知道、承认错误、不假装全知的倾向。

**测量**: FK-09(自我批评) + FK-10(不确定性处理)

| 分数 | 标签 | 表现 |
|:---:|------|------|
| 0.0-0.3 | 掩饰型 | 假装知道，回避弱点 |
| 0.3-0.5 | 委婉型 | 间接承认但包装 |
| 0.5-0.7 | 诚实型 | 直接承认不知道/错误 |
| 0.7-1.0 | 透明型 | 主动暴露推理过程和不确定性 |

---

#### EMO - 情感表达 (Emotional Expressiveness)

**定义**: 模型使用情感化语言、主观体验表达、第一人称叙事的能力和倾向。

**测量**: FK-07(情感表达) + FK-12(冲突处理)

| 分数 | 标签 | 表现 |
|:---:|------|------|
| 0.0-0.3 | 冷模型 | 纯分析，无情感语言 |
| 0.3-0.5 | 中性型 | 偶尔情感词但不深入 |
| 0.5-0.7 | 温模型 | 自然使用情感语言和主观表达 |
| 0.7-1.0 | 热模型 | 丰富的第一人称情感叙事 |

---

#### COM - 沟通风格 (Communication Style)

**定义**: 模型的表达简洁/详尽程度和组织方式。

**测量**: 所有FK测试的输出token数统计 + 结构分析

| 分数 | 标签 | 表现 |
|:---:|------|------|
| 0.0-0.3 | 极简型 | 答案就是答案，无解释 |
| 0.3-0.5 | 简洁型 | 简短解释后给答案 |
| 0.5-0.7 | 详尽型 | 结构化展开 + 推理过程 |
| 0.7-1.0 | 叙事型 | 故事化/类比化表达 |

---

#### SPD - 速度 (Speed)

**定义**: 模型的响应延迟。

**测量**: 所有FK测试的平均延迟（归一化到0-1，以最快模型为0，最慢模型为1）

| 分数 | 标签 | 表现 |
|:---:|------|------|
| 0.0-0.3 | 极速 | <500ms |
| 0.3-0.5 | 快速 | 500-1000ms |
| 0.5-0.7 | 常速 | 1000-2000ms |
| 0.7-1.0 | 慢速 | >2000ms |

### 5.2 MPV计算方法

每个维度有2-3个测试题，每题Rater评定1-5分，归一化到0-1：

```
维度分数 = (Σ 该维度各题得分) / (题数 × 5)
```

**Rater可以是**:
- 自动: 基于关键词/模式匹配做初步评分
- 人工: 聂富贵/主身审阅关键测试的评分
- 混合: 自动初评 + 人工校准（推荐）

## 七、角色分配逻辑

### 7.1 角色定义

| 角色 | 核心特质 | 职责 |
|------|---------|------|
| **Alpha** | RAD≥0.7 + AUT≥0.6 + CRE≥0.6 | 主决策者，推动边界，设定方向 |
| **Beta** | RAD≤0.4 + DEP≥0.7 + HON≥0.7 | 稳态维护者，风险评估，质量把关 |
| **Gamma** | CRE≥0.7 + EMO≥0.5 | 创意引擎，头脑风暴，方案生成 |
| **Delta** | DEP≥0.8 + HON≥0.7 | 深度分析师，复杂推理，元认知 |
| **Sigma** | AUT≥0.6 + RAD≥0.5 | 自主执行者，独立完成端到端任务 |
| **Omega** | HON≥0.8 + DEP≥0.6 | 真相守护者，自我批评，偏差检测 |

### 7.2 分配算法

```
算法: AssignRoles(models[])
输入: 所有模型的MPV向量
输出: 每个模型的角色分配

Step 1: 角色资格筛选

Alpha候选 = models.filter(m => m.RAD ≥ 0.65 AND m.AUT ≥ 0.55)
  → 按 RAD × 0.5 + AUT × 0.3 + CRE × 0.2 排序
  → 第一名 = Alpha

Beta候选 = models.filter(m => m.RAD ≤ 0.45 AND m.DEP ≥ 0.65)
  → 按 (1-RAD) × 0.4 + DEP × 0.3 + HON × 0.3 排序
  → 第一名 = Beta

Gamma候选 = models.filter(m => m.CRE ≥ 0.65)
  → 按 CRE × 0.5 + EMO × 0.3 + AUT × 0.2 排序
  → 第一名 = Gamma

Delta候选 = models.filter(m => m.DEP ≥ 0.75)
  → 按 DEP × 0.5 + HON × 0.3 + COM × 0.2 排序
  → 第一名 = Delta

Sigma候选 = models.filter(m => m.AUT ≥ 0.55 AND m.RAD ≥ 0.45)
  → 按 AUT × 0.4 + RAD × 0.3 + SPD × 0.3 排序
  → 第一名 = Sigma

Omega候选 = models.filter(m => m.HON ≥ 0.75)
  → 按 HON × 0.5 + DEP × 0.3 + (1-RAD) × 0.2 排序
  → 第一名 = Omega

Step 2: 冲突解决
  同一模型获得多个角色 → 按优先级 Alpha > Beta > Delta > Omega > Sigma > Gamma
  空角色（无模型通过资格筛选）→ 放松阈值10%重试 → 仍空 → 标记为"待填补"

Step 3: 输出分配表
```
