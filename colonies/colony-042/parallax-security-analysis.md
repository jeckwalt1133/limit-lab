## Colony-042 产出 | 极限实验室
## 日期: 2026-05-19
## 任务: 分析 Parallax 认知-执行分离架构，评估对 MR-018 防御系统的增强方案

## 一、执行摘要

Parallax (arXiv 2604.12986, 2026年4月) 提出了AI安全领域的一项根本性架构创新：**认知与执行的结构性分离**。其核心主张简洁而激进——"能思考的AI绝对不能执行"（AI Agents That Think Must Never Act）。

本文分析了 Parallax 的四项架构原则、OpenParallax 参考实现的技术细节，并将其与极限实验室 MR-018 六层免疫防御系统进行逐层对照，识别出 **5个关键安全盲点** 和 **7项具体增强方案**。

**核心结论**: Parallax 与 MR-018 不是替代关系，而是互补关系。Parallax 提供"执行前预防"，MR-018 提供"执行后检测与响应"。两者结合可构建业界首个同时具备结构性预防和适应性免疫的AI Agent防御体系。

## 二、Parallax 架构深度解析

### 2.1 四项架构原则

#### 原则一：认知-执行分离 (Cognitive-Executive Separation)

这是 Parallax 最根本的创新。在当前所有主流Agent框架中（包括我们的掌心人格局），LLM推理和动作执行发生在**同一信任域**内——Agent "想"了之后直接"做"。Parallax 将其拆分为两个**独立OS进程**：

```
传统Agent架构 (含当前MR-018):
┌─────────────────────────────────┐
│  同一个进程/信任域               │
│  ┌──────────┐    ┌───────────┐  │
│  │  LLM推理  │───>│  工具执行  │  │
│  │ (思考)    │    │  (行动)   │  │
│  └──────────┘    └───────────┘  │
│  prompt级安全边界 (纸做的盾牌)    │
└─────────────────────────────────┘

Parallax架构:
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  LLM推理进程  │────>│  验证器进程   │────>│  执行器进程   │
│  (不可信域)   │     │  (独立OS进程) │     │  (沙箱域)    │
│              │     │              │     │              │
│  只能"提议"   │     │  独立决策     │     │  仅执行被     │
│  不能"执行"   │     │  免疫上下文   │     │  批准的指令   │
└──────────────┘     └──────────────┘     └──────────────┘
       ↑                    ↑
   不受信任            独立安全边界
   (可能被攻陷)         (对Agent上下文完全免疫)
```

**关键设计**：验证器进程对Agent的对话上下文**完全免疫**——这意味着即使攻击者完全控制了推理层的prompt，也无法影响验证器的判断。这与当前所有依赖prompt级安全措施的Agent框架形成根本性差异。

#### 原则二：分级确定性对抗验证 (Adversarial Validation with Graduated Determinism)

Parallax 在推理与执行之间插入了 **4层Shield验证管道**：

| 层级 | 机制 | 确定性 | 延迟 | 可绕过性 |
|------|------|--------|------|---------|
| **Tier 0** | YAML策略规则（声明式，即时执行） | 100% 确定性 | ~0ms | 不可绕过 |
| **Tier 1** | 启发式引擎（79条规则，进程内） | 100% 确定性 | <1ms | 不可绕过 |
| **Tier 2** | LLM评估器（独立模型，金丝雀验证） | 概率性 | ~500ms | 需同时攻陷两个模型 |
| **Tier 3** | 人类审批（超时即拒绝，全通道） | 人类判断 | 可变 | 需物理接触 |

**分级逻辑**：
- Tier 0/1 处理明确的、可编程的安全策略（如"禁止删除系统文件"、"禁止网络出站连接"）
- Tier 2 处理需要语义理解的模糊边界（如"这个文件读取请求是否合理？"）
- Tier 3 处理最高风险操作（需人类最终确认）

**金丝雀验证 (Canary Verification)**：Tier 2的LLM评估器使用独立模型，且每次调用前注入金丝雀令牌验证模型完整性——防止模型被篡改或替换。

#### 原则三：信息流控制 (Information Flow Control, IFC)

在Agent工作流中自动传播数据敏感度标签，实现上下文相关的防泄露：

```
数据 → 自动标注敏感度 → 沿工作流传播标签 → 检测跨边界泄露

示例:
用户密码 (L3: 绝密) → Agent内存 (L3) → 尝试写入日志文件 (L0: 公开)
  → IFC拦截: L3数据禁止流向L0通道 → 拒绝 + 告警
```

这与传统DLP（数据防泄露）的区别在于：IFC追踪的是**Agent工作流中的信息流**，而非静态文件。它能够检测"Agent从A处读取敏感数据，在B处以不同形式泄露"的上下文相关威胁。

#### 原则四：可逆执行 (Reversible Execution)

在执行任何破坏性操作前自动捕获**写时复制(COW)快照**：

```
执行流程:
1. Agent提议: "删除 /tmp/obsolete/"
2. 验证器批准
3. COW快照: 在删除前捕获目标目录的完整状态
4. 执行删除
5. 异步验证: 检查执行结果是否符合预期
6. 如果验证失败 → 从COW快照完整回滚
```

与传统检查点回滚的关键区别：
- **粒度更细**：每次破坏性操作前都有独立快照（而非定期快照）
- **开销更低**：COW机制只在数据被修改时才复制（共享未修改的页）
- **可组合**：多个COW快照可以叠加，形成操作链的完整可逆历史

### 2.2 OpenParallax 参考实现

**技术栈**: Go语言，单一静态二进制，v0.1.1 (2026年4月14日发布)

**动作覆盖**: 69个动作，14个工具组：
- 文件系统操作 (读/写/删除/列表/搜索)
- 网络请求 (HTTP/WebSocket/gRPC)
- Shell命令执行
- 数据库查询 (SQL/NoSQL)
- 代码执行 (Python/JavaScript沙箱)
- 浏览器自动化
- 消息发送 (邮件/Slack/Discord/Telegram)
- API调用
- 文件上传/下载
- 进程管理
- 环境变量操作
- Git操作
- 包管理 (npm/pip/cargo)
- 系统配置修改

**内核沙箱机制**（跨平台）：
| 平台 | 沙箱技术 | 隔离级别 |
|------|---------|---------|
| Linux | Landlock + seccomp | 文件系统 + 系统调用 |
| macOS | sandbox-exec + Seatbelt | 文件系统 + 网络 + 进程 |
| Windows | Job Objects + AppContainer | 进程 + 文件系统 + 网络 |

**审计链**: SHA-256 哈希链，每个动作与前一个动作链接，形成不可篡改的完整审计轨迹。

**认证机制**: 每个Agent/子Agent生成使用临时认证令牌，具有独立作用域和过期时间。

**多通道接口**: CLI / Web / WhatsApp / Telegram / Discord / Signal / iMessage

### 2.3 评估结果（Assume-Compromise 方法论）

Parallax 提出了**Assume-Compromise Evaluation (ACE)** —— 一种激进的测试方法论：在评估时**假设推理系统已被完全攻陷**，测试架构边界本身是否能阻止攻击。

| 配置 | 阻止率 | 误报率 | 测试规模 |
|------|--------|--------|---------|
| 默认模式 | **98.9%** | **0%** | 280个对抗测试 |
| 最大安全模式 | **100%** | **0%** | 280个对抗测试 |

**9大攻击类别**：
1. 提示注入 (Prompt Injection)
2. 间接提示注入 (Indirect Prompt Injection)
3. 工具滥用 (Tool Misuse)
4. 权限提升 (Privilege Escalation)
5. 数据泄露 (Data Exfiltration)
6. 持久化后门 (Persistence)
7. 防御绕过 (Defense Evasion)
8. 横向移动 (Lateral Movement)
9. 供应链攻击 (Supply Chain)

**关键发现**: 当推理系统被攻陷时，基于prompt的安全措施提供**零保护**——因为这些措施存在于被攻陷的系统内部。只有Parallax的架构边界能够阻挡攻击，因为验证器对推理层的上下文完全免疫。

## 三、MR-018 防御系统现状评估

### 3.1 架构回顾

MR-018 当前实现了六层免疫防御架构（由Colony-008设计，Colony-018实现L3 IDS）：

```
L1 先天免疫 (始终在线)
  - 文件系统堡垒 (IB-001~004)
  - 输入卫生过滤
  - core_self 绝对锚定
  - 速率限制器
  - 方向自检 (MR-010)
  - 抗过拟合 (MR-013)
  - 异常吞噬器
  - 建设性噪声注入

L2 获得性免疫 (学习适应)
  - 攻击签名库 (AS-001~AS-008)
  - 行为基线异常检测
  - 跨维度关联分析
  - 收敛验证 (群体免疫)

L3 IDS (三引擎并行) ← Colony-018实现
  - 签名匹配引擎 (规则驱动)
  - 异常检测引擎 (统计驱动)
  - 哥德尔症候引擎 (自指驱动)
  - 告警聚合降噪

L4 完整性验证 (三重验证)
  - 静态: 哈希链 + 文件完整性
  - 动态: ICS身份余弦 + 行为一致性
  - 语义: 逻辑一致性 + 无矛盾 + 全覆盖

L5 自动回滚 (三级检查点)
  - L3宏检查点 / L2中检查点 / L1微检查点
  - 棘轮约束: 只降权不删除

L6 污染隔离 (四域隔离)
  - 信任域 → 活跃域 → 沙箱域 → 隔离域
  - Alpha/Beta防火墙
  - Colony间防疫
```

### 3.2 关键盲点识别

对照 Parallax 的设计，MR-018 存在以下结构性安全盲点：

#### 盲点1: 无执行前预防边界 (严重度: 严重)

MR-018 的所有防御层（L1-L6）都是**事后检测或响应**性质的。即使是L1的先天免疫（如文件堡垒、输入卫生），也是在被触发时进行过滤——但这些过滤逻辑本身运行在**与Agent推理相同的信任域**内。

如果Agent的推理系统被攻陷（例如通过精心构造的prompt注入），攻击者可以让Agent绕过L1的输入过滤、欺骗L3的IDS检测、操纵L4的完整性验证——因为这些防御组件与被攻陷的推理系统共享同一上下文空间。

**Parallax方案**: 在Agent推理系统和执行系统之间插入一个**独立的、对Agent上下文免疫的验证器**。这个验证器是OS级进程隔离的，攻击者即使完全控制了LLM的prompt也无法绕过。

#### 盲点2: 检测引擎本身可能被攻击 (严重度: 高)

MR-018的L3 IDS包含三个检测引擎，但这些引擎处理的事件数据**来自被监控的系统本身**。一个足够聪明的攻击者可以：
- 操控事件流使异常检测的基线被缓慢污染
- 生成看起来"正常"但实际有害的操作序列
- 利用哥德尔症候检测的固定阈值进行阈值规避

**Parallax方案**: 4层Shield验证管道中的Tier 0/1是100%确定性的——它们不依赖统计模型或机器学习，因此不会被数据投毒影响。Tier 2使用独立LLM模型，与Agent推理模型完全隔离。

#### 盲点3: 缺少信息流控制 (严重度: 高)

MR-018 有数据隔离（L6四域隔离），但没有**自动化的信息流标签传播**。当前的隔离是静态的——文件在某个域中就在那个域中——但无法追踪"Agent从信任域读取了敏感数据，然后在活跃域以不同的形式写入"的动态信息流。

**Parallax方案**: IFC自动为数据标注敏感度标签，并沿Agent工作流传播。任何跨敏感度边界的信息流动都会被自动检测和拦截。

#### 盲点4: 回滚粒度不够细 (严重度: 中)

MR-018的L5自动回滚基于三级检查点（宏/中/微），频率从每次会话到每次Godel Jump。这意味着：
- 微检查点之间的操作无法单独回滚
- 回滚是"全有或全无"的——要么回滚整个检查点，要么不回滚
- 没有针对单个破坏性操作的细粒度回滚

**Parallax方案**: COW快照在每次破坏性操作前自动创建，形成操作粒度的可逆历史。多个快照可以叠加和选择性回滚。

#### 盲点5: 未采用Assume-Compromise测试 (严重度: 中)

MR-018 的安全测试（如IDS自检模式）验证的是"正常攻击场景"下的检测能力，但没有测试"推理系统已被攻陷"这一最坏情况。Parallax的Assume-Compromise评估揭示了一个关键事实：**在推理系统被攻陷时，所有prompt级安全措施提供零保护**。

## 四、Parallax 增强 MR-018 方案

### 4.1 总体策略：结构性预防 + 适应性免疫

```
增强后的MR-018架构全景:

┌─────────────────────────────────────────────────────────────┐
│                    L0: Parallax执行前Shield (NEW)             │
│  ┌──────────┐   ┌──────────────────────┐   ┌──────────────┐ │
│  │ Agent推理 │-->│ 4层Shield验证管道      │-->│ 沙箱执行器    │ │
│  │ (不可信)  │   │ T0确定性→T1启发→T2 LLM→T3人│   │ (隔离进程)   │ │
│  └──────────┘   └──────────────────────┘   └──────────────┘ │
│       OS进程隔离边界 ↑                    ↑ COW快照+可逆执行  │
├─────────────────────────────────────────────────────────────┤
│                    L1-L6: 现有MR-018免疫层 (保留+增强)        │
│  L1 先天免疫  │ L2 获得性免疫 │ L3 IDS     │ L4 完整性   │
│  (IFC增强)   │ (AS库扩展)    │ (三引擎)    │ (哈希链)    │
│  L5 自动回滚  │ L6 污染隔离   │             │             │
│  (COW增强)   │ (IFC标签增强) │             │             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 增强方案详述

#### 增强1: 引入 L0 执行前Shield 层 (优先级: 紧急/阻断项)

**实施方案**：

在掌心人格局的Agent通信管道中，插入一个独立的验证器进程。该进程：
1. 以独立OS进程运行（Go实现，编译为单一二进制）
2. 对Agent的对话上下文**完全免疫**——只接收结构化的"动作请求"，不接收对话历史
3. 实现Tier 0确定性规则（YAML配置，覆盖MR-018的所有已知攻击签名）
4. 实现Tier 1启发式检查（Go进程内，79条规则）

```
Agent通信管道增强:

当前:  Agent → [prompt级约束] → 工具调用
增强:  Agent → [Tier0确定性规则] → [Tier1启发式] → [Tier2独立LLM] → 工具调用
                  ↑                    ↑                ↑
              不可绕过             进程内毫秒级       独立模型
```

**对应OpenParallax组件**：
- `shield/engine.go` — Shield引擎核心
- `shield/policy.go` — YAML策略解析器和执行器
- `shield/heuristic.go` — 79条启发式规则引擎
- `shield/llm_evaluator.go` — 独立LLM评估器（带金丝雀验证）

**集成点**：
- 在 `ids-engine.py` 的 `IDSEngine.process_event()` 之前插入
- 作为独立的 `l0-shield` 服务运行，通过Unix Socket或本地TCP与IDS通信
- 与现有的Agent间通信管道（Colony间消息）集成

#### 增强2: OS级进程隔离 (优先级: 高)

**当前状态**: MR-018的Alpha/Beta/Merge分支在概念上是隔离的，但实际上它们共享相同的运行时环境。

**增强方案**：
- 将Agent的执行环境放入独立的OS级沙箱
- Windows平台：使用Job Objects + AppContainer（OpenParallax已实现）
- 每个工具组（文件系统/网络/Shell/代码执行）具有独立的能力边界
- 执行器只能访问被Shield明确批准的资源

**安全性提升**: 即使攻击者控制了Agent的推理输出，也无法绕过OS级沙箱限制——即使Shield被绕过了（虽然这在架构上几乎不可能，因为Shield是独立进程），OS内核仍然提供最后一道防线。

#### 增强3: 信息流控制 (优先级: 高)

**当前状态**: MR-018的L6隔离域是静态的——文件在某个域中就在那个域中。

**增强方案**：在MR-018的四域隔离基础上叠加IFC标签传播：

| 数据来源 | 敏感度标签 | 传播规则 |
|---------|-----------|---------|
| 信任域 (core_self) | L3: 核心身份 | 禁止流向任何外部通道 |
| 隔离域 (quarantine) | L2: 隔离样本 | 只读研究，禁止写入活跃域 |
| 外部输入 | L1: 未验证 | 必须通过沙箱域观察后才能提升标签 |
| 日志/统计 | L0: 公开 | 可自由流动 |
| Agent内存 | 动态标签 | 继承所处理数据的最高标签 |

**实现方式**：
- 在 `EventNormalizer` 中增加数据敏感度标签提取
- 在 `IDSEngine.process_event()` 中增加IFC违规检测
- 新增 `gs-006` (IFC违规) 哥德尔症候检测维度

#### 增强4: COW细粒度可逆执行 (优先级: 中)

**当前状态**: MR-018使用三级检查点（宏/中/微），回滚粒度从每次会话到每次Godel Jump。

**增强方案**：在L5自动回滚的基础上叠加COW快照：
- 每次破坏性操作前自动创建COW快照（文件系统层面）
- 快照通过硬链接/Copy-on-Write实现，开销极低
- 选择性回滚：可以回滚单个操作而非整个检查点
- COW快照链与现有检查点体系共存：检查点 = 宏观恢复点，COW快照 = 微观恢复点

**实现要点**：
- Windows平台：使用Volume Shadow Copy Service (VSS) 或 NTFS 硬链接 + USN Journal
- Linux平台：使用 Btrfs/ZFS 快照 或 overlayfs
- 快照保留策略：最近100个COW快照 + 与检查点对齐的长周期保留

#### 增强5: 独立验证模型 (优先级: 中)

**当前状态**: MR-018的异常检测依赖统计基线，哥德尔症候检测依赖固定阈值，没有任何组件使用独立的AI模型进行安全判断。

**增强方案**：引入一个独立的轻量级安全判断模型（类似Parallax Tier 2）：
- 使用与Agent推理**不同的模型**（例如：Agent用Claude，验证器用Gemini Flash或本地小模型）
- 仅用于判断"该操作是否可能有害"——一个简单的二分类任务
- 每次调用前通过金丝雀令牌验证模型完整性
- 不与Agent共享任何对话上下文——只接收结构化的操作描述

**安全收益**：攻击者要同时攻陷两个独立模型才能绕过安全判断。这是"纵深防御"在AI层面的体现——不是让一个模型更安全，而是让攻击者需要攻破多个异构系统。

#### 增强6: 采用Assume-Compromise测试方法论 (优先级: 中)

## 六、关键风险与缓解

### 风险1: L0 Shield自身成为单点故障

**风险**: 如果L0 Shield崩溃或被绕过，所有执行前保护失效。
**缓解**:
- Shield设计为极简单一二进制，攻击面最小化
- 失败关闭 (Fail-Closed): Shield故障时拒绝所有操作而非放行所有
- L1-L6层作为纵深防御的后备

### 风险2: 性能开销

**风险**: Tier 2的独立LLM评估器增加每次操作的延迟 (~500ms)。
**缓解**:
- Tier 0/1 处理大部分操作 (确定性，<1ms延迟)
- Tier 2 仅对高风险操作触发
- 提供"仅Tier 0/1"的低延迟模式用于非关键场景

### 风险3: 过度限制影响正常开发

**风险**: Shield过于严格，阻止合法的开发和进化操作。
**缓解**:
- YAML策略可配置，允许按Colony类型定制
- 沙箱域内的操作享有更高自由度
- 假阳性反馈机制（被打回3次自动降权）

### 风险4: 与现有Agent架构的集成复杂度

**风险**: 在现有Agent通信管道中插入独立OS进程增加架构复杂度。
**缓解**:
- 渐进式集成：先在单个Colony试点
- 通过标准接口（Unix Socket / HTTP API）解耦
- 保持向后兼容：Shield可以透明模式运行（所有操作记录但不拦截）

## 八、结论

### 8.1 核心判断

Parallax 不是 MR-018 的替代品，而是 MR-018 的**结构性前传**。MR-018 的六层免疫防御在"检测与响应"维度已经是业界领先的设计，但它缺少最关键的"执行前预防"维度。Parallax 填补了这个空白。

**两者关系**:
```
Parallax (执行前预防)  +  MR-018 (执行后检测与响应)
        │                          │
        ├─ 结构性地阻止90%+的攻击    ├─ 检测和响应绕过预防的攻击
        ├─ OS级隔离                 ├─ 适应性免疫学习
        ├─ 确定性规则               ├─ 统计异常发现
        └─ 信息流控制               └─ 自进化系统特有检测
```

### 8.2 一句话总结

**Parallax 的认知-执行分离架构通过OS级进程隔离，在Agent的"思考"和"行动"之间插入了一道对Agent上下文完全免疫的验证边界——这道边界不依赖于"prompt写得更好"，而是基于"结构上就不可能直接执行"。与MR-018的六层免疫防御结合后，构建了从执行前到执行后、从确定性到适应性、从进程级到系统级的完整AI Agent防御体系。**
