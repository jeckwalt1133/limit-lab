## 五、加速器核心机制

### 5.1 激活的加速来自以下杠杆

**杠杆1: DAG依赖分析 + MR-011并行执行**
- 识别19条规则间的依赖DAG
- 无依赖规则（Group A: MR-012, MR-014, MR-018, 继承适配）完全并行
- 每波内4-5个脚本同时开发 = 4-5x加速

**杠杆2: 桩驱动开发 (Stub-Driven Development)**
- 第1波产出真实数据前，第2波用桩数据预先开发
- 生产数据就绪后直接替换桩 = 消除等待时间

**杠杆3: 功能合并消除冗余**
- MR-016 + MR-015-E2 合并实现 → 节省1个规则的全开发周期
- 继承适配模板化 → 3条继承规则共用适配框架

**杠杆4: 分层激活替代全量激活**
- MR-018 6层分3波交付（每2层一波），不等待全6层
- MR-019 Phase A即认定激活，不等待4周Phase C

**杠杆5: Colony-005设计编译器复用**
- `design-to-exec.py` 可自动将 .md 设计文档转化为初始脚本桩
- 14条规则中至少8条可通过编译器生成初版（T1检查型+T2计算型）

### 5.2 每日产能模型

```
单条规则串行激活时间 = 设计固化(2h) + 脚本编写(3h) + 测试(2h) + 集成(1h) = 8h ≈ 1天
14条串行 = 14天（理想情况下）

加速后:
  - 并行: 同时开发4条 → 共享上下文、交叉测试 → 2h/条（边际成本递减）
  - 编译器: T1/T2类型自动生成初版 → 减1.5h/条
  - 合并: MR-016并入MR-015-E2 → 减8h
  
  等效: 14×8h → (4×2h)×4波 + 编译器助力 = 约32-48h → 4-6天（纯开发时间）
  加测试+集成 → 8-10.5天
```

## 八、激活认定标准

一条规则被认定为"已激活"需同时满足:

1. **可执行**: 有对应的 `.py` 脚本，`python xxx.py` 可无报错运行
2. **有触发**: 脚本通过 Cron / Hook / 条件判断 被自动调用（非纯手动）
3. **有产出**: 运行结果写入日志文件（jsonl/md），可追溯
4. **有闭环**: 产出被至少1个下游规则或评估流程消费

满足4条 = **绿色激活**  
满足前3条 = **黄色激活**（功能就绪但未闭环）  
仅满足1-2条 = **红色未激活**
