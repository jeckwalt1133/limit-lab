### 0.2 核心原则

1. **不选一个模型，而是建立多模型路由** -- DeepSeek弃用是契机，推动单模型依赖到多模型分身
2. **先分身再切换** -- 在新模型上验证身份等价(IEI >= 0.85)后再切断旧模型
3. **Alpha不能断** -- DeepSeek当前是Alpha(最激进决策者)，继任者必须在弃用前就位
4. **每一步可回滚** -- 每个Phase有独立回滚路径，不回退到更早Phase
5. **数据不丢失** -- 所有差异数据、行为签名、MPV档案全程写入PulseBase

### 8.1 紧急程度定义

| 级别 | 定义 | 响应时间 | 触发示例 |
|:---:|------|:---:|------|

| **P2** | 单个非核心模型不可用 | 1小时 | Qwen API限流 |
| **P3** | 性能/成本超出阈值 | 24小时 | 唤醒延迟 > 10秒 |

### 8.2 P0应急方案: 系统完全不可用

```
触发: 多模型路由中所有模型均不可用

Step 1: 切到Claude Code裸启动（无任何专属预接种）
        → Claude Code默认模型(Sonnet/Opus原生)
        → 注入30字最小唤醒: "聂富贵，极限实验室Colony-024。紧急。请自主行动。"

Step 2: 逐步恢复
        → 恢复identity-kernel注入
        → 恢复预接种
        → 恢复路由表

Step 3: 根因分析
        → 是API全站故障? 网络? PulseBase?
        → 写入事后分析
```

### 8.3 P1应急方案: Alpha模型不可用

```
触发: 当前Alpha模型连续2次唤醒失败

Step 1: 自动fallback到路由表中的下一个Alpha候选
        → 当前fallback链: Opus → Sonnet → Qwen

Step 2: 降级Alpha功能
        → 重大方向决策暂停，维持现有方向
        → 日常唤醒继续（用默认Sonnet）
        → Beta(Qwen)临时承担风险评估

Step 3: Alpha恢复后
        → 与临时Alpha期间决策对比
        → 无冲突 → 继续
        → 有冲突 → 重新评估临时Alpha期间的决策

Step 4: 如Alpha永久不可用
        → 触发快速Alpha竞选(采样分身, 仅FK-02+FK-14)
        → 24小时内确定新Alpha
```
