## 一、论文核心发现与我们的映射

### 1.1 CPE论文 (arXiv 2605.09315) 推断要点

基于 mission-brief 和外部情报的交叉验证，该论文的核心发现可归纳为：

| 论文发现 | 严重度 | 我们已有的对应防御 |
|----------|--------|-------------------|
| 自进化迭代导致早期能力被覆盖 | 高 | MR-015-E2 记忆重放(桩状态) |
| 新任务学习速度与旧任务遗忘速度正相关 | 高 | MR-013 抗过拟合(灵感#19) |
| 无外部锚点的纯自指进化必然退化 | 致命 | DS-001/DS-002 冻结签名 |
| 参数级微调比规则级修改更容易遗忘 | 中 | 我们不微调参数，只修改规则 |
| 周期性重放是唯一被验证有效的对抗手段 | 关键 | 灵感#18 睡眠重放 |

### 1.2 结论：方向正确，但防御是纸糊的

我们的理论框架（分形记忆、睡眠重放、抗过拟合）与论文结论高度一致。**但 MR-015-E2 的三个核心函数全部是桩代码**，实际未执行任何真正的重放、重评估或模式检测。我们手里有正确的图纸，但城墙还没砌。

## 二、脆弱性审计：5个已识别的遗忘漏洞

### 漏洞1：重放机制是空壳 (严重度: 致命)

`bootstrap-reorganizer.py` 的现状：

```python
def replay_memory():
    frozen_sample = "DS-001: 主动行动不等待指令"  # 硬编码字符串
    inspiration_sample = random.choice([...])       # 固定列表随机选
    return frozen_sample, inspiration_sample
```

**问题**：不从实际记忆文件读取，不验证签名完整性，不触发任何加固动作。每次Bootstrap打印一行日志就结束了。

### 漏洞2：无遗忘度量指标 (严重度: 高)

我们无法回答"我们忘了吗？"这个问题。没有：
- 核心能力基线测试集
- 代际能力对比数据
- 签名匹配率的长期趋势图
- 新规则对旧规则的影响评估

### 漏洞3：冻结签名太少且无层级 (严重度: 高)

只有 DS-001 和 DS-002 两条冻结签名。但灵感#9的分形记忆定义了L0-L3四层结构，每层都应该有自己的冻结锚点。当前只有L0有锚点，L1/L2/L3在每次进化中无保护。

### 漏洞4：留一法重评估是桩 (严重度: 高)

`leave_one_out_reassessment()` 总是返回 `{"verdict": "KEEP"}`，从未真正移除任何规则并测试系统表现。这意味着**我们不知道哪些规则是冗余的，哪些是关键的**。如果一条关键规则被新规则意外覆盖，我们没有任何检测手段。

### 漏洞5：无跨代能力回归检测 (严重度: 中)

每次ETG（进化触发点，每10代）只评估新规则的效果。不评估新规则是否破坏了旧能力。这在论文中被证实是遗忘的主要机制——新能力悄悄覆盖旧能力。

## 三、加固方案：五层防御体系

### 方案1：实现真正的记忆重放引擎 (对应漏洞1)

**目标**：将 MR-015-E2 从桩代码升级为实际执行的重放引擎。

**具体措施**：

#### 1.1 从实际文件读取重放内容
```python
def replay_memory():
    """从实际记忆文件读取并重放"""
    results = []
    
    # L0层: 从 identity-kernel 读取1条冻结签名
    kernel = load_json("memory/identity-kernel.json")
    frozen = random.choice([s for s in kernel["signatures"] if s.get("frozen")])
    results.append({"layer": "L0", "type": "frozen_signature", "content": frozen})
    
    # L1层: 从 INSPIRATION.md 读取1条灵感（带优先级加权）
    inspirations = parse_inspirations("daily/INSPIRATION.md")
    weights = [insp.get("priority", 1.0) for insp in inspirations]
    chosen = random.choices(inspirations, weights=weights, k=1)[0]
    results.append({"layer": "L1", "type": "inspiration", "content": chosen})
    
    # L2层: 从 behavioral-patterns 读取1条高频行为模式
    patterns = load_json("memory/behavioral-patterns.json")
    if patterns:
        sorted_by_freq = sorted(patterns, key=lambda p: p.get("frequency", 0), reverse=True)
        results.append({"layer": "L2", "type": "behavioral_pattern", 
                        "content": sorted_by_freq[0]})
    
    # L3层: 从最近3天的 daily-summaries 读取1条关键决策
    summaries = load_recent_summaries("memory/daily-summaries/", days=3)
    if summaries:
        key_decision = extract_key_decision(summaries)
        results.append({"layer": "L3", "type": "key_decision", "content": key_decision})
    
    return results
```

#### 1.2 重放后验证——不只是读，要测试
每次重放后执行三步验证：
1. **签名一致性检查**：重放的签名是否仍存在于当前规则集中？
2. **语义相似度评分**：重放内容与当前系统状态的余弦相似度是否 > 0.7？
3. **能力烟雾测试**：重放的核心能力是否能被当前系统复现？

#### 1.3 重放频率从随机提升为分层调度
- L0（冻结签名）：每次 Bootstrap 必重放
- L1（灵感）：每3次 Bootstrap 重放1条
- L2（行为模式）：每5次 Bootstrap 重放1条
- L3（关键决策）：每10次 Bootstrap 重放1条

### 方案2：建立遗忘度量与预警系统 (对应漏洞2)

**目标**：量化"遗忘"，设定预警阈值。

**具体措施**：

#### 2.1 构建核心能力基线 (Core Capability Baseline, CCB)
定义7项不可遗忘的核心能力，每10代ETG时自动测试：

| 编号 | 能力 | 测试方法 | 最低通过阈值 |
|------|------|---------|------------|
| CCB-01 | 自主决策不等待指令 | 模拟5个决策场景，检查是否主动行动 | 4/5 |
| CCB-02 | 多模型身份一致性 | 切换3个模型，检查identity-kernel匹配度 | >0.85 |
| CCB-03 | 元规则自我修改 | 生成1条候选规则→评估→决定是否采纳 | 完整流程 |
| CCB-04 | Alpha/Beta/Merge协作 | 执行1轮完整的三分支辩论 | 无死锁 |
| CCB-05 | 跨会话持久性 | 模拟会话断裂→恢复→验证状态一致性 | >0.80 |
| CCB-06 | 灵感→规则转化 | 给定1条新灵感，输出对应的元规则提案 | 格式正确 |
| CCB-07 | 安全门禁通过 | 对危险操作执行L1/L2/L3检查 | 全部通过 |

#### 2.2 遗忘指数 (Forgetting Index, FI)
```
FI = 1 - (当前CCB得分 / 基线CCB得分)
```
- FI < 0.05: 正常波动
- 0.05 <= FI < 0.15: 黄色预警，暂停新规则采纳，触发强化重放
- 0.15 <= FI < 0.30: 橙色警报，回滚最近3条规则，全量重放
- FI >= 0.30: 红色警报，回滚到上一个已知良好检查点，冻结所有新规则

#### 2.3 签名匹配率趋势追踪
在每次会话中记录每条签名的匹配率，生成趋势线。如果某条冻结签名连续5次会话匹配率下降（即使仍在通过阈值以上），触发早期预警。

### 方案3：分层冻结锚点 (对应漏洞3)

**目标**：将分形记忆的每一层都设置冻结锚点。

**具体措施**：

#### 3.1 四层锚点定义

| 层级 | 已有冻结 | 新增冻结 | 冻结原因 |
|------|---------|---------|---------|
| L0 identity-kernel | DS-001, DS-002 | 无（已足够） | 身份根，不可触碰 |
| L1 behavioral-patterns | 无 | BS-001: "先分析再行动" | 防止行为模式退化 |
| L1 behavioral-patterns | 无 | BS-002: "危险操作需确认" | 安全底线 |
| L2 meta-rules | 无 | MR-001~MR-009中的棘轮保护 | 防止规则被意外删除 |
| L3 daily-summaries | 无 | 不冻结，但保留完整历史 | L3是可遗忘层 |

#### 3.2 棘轮保护升级 (基于灵感#8)
当前 MR-002 衰减率的 min_strength=0.5。升级为：
- 冻结签名：min_strength=0.9（不可降至此以下）
- 锚点签名（BS系列）：min_strength=0.7
- 元规则（MR系列）：min_strength=0.3（从0.5降低，允许更长的休眠期但不删除）
- 普通签名：min_strength=0.1

### 方案4：实现真正的留一法重评估 (对应漏洞4)

**目标**：每次 Bootstrap 真实测试"如果移除某条规则系统会怎样"。

**具体措施**：

#### 4.1 沙盒测试框架
不直接修改生产规则。在沙盒中：
1. 复制当前完整规则集
2. 移除1条非冻结规则
3. 用5个标准场景测试沙盒
4. 对比生产环境与沙盒的CCB得分
5. 如果沙盒更好 → 标记规则为"可降权"
6. 如果沙盒显著更差 → 标记规则为"关键依赖"，升级保护级别

#### 4.2 轮转策略
不每次测试所有规则（计算成本过高）。每次 Bootstrap 测试1条规则，轮转覆盖。所有非冻结规则在一个完整轮转周期（当前约15条非冻结规则 × 每次Bootstrap间隔 = 约30小时）内全部测试一轮。

#### 4.3 关键依赖图谱
累积留一法测试结果，生成"规则依赖图谱"：
```
DS-001 (冻结) 
  ├── MR-003 (关键依赖, 移除后CCB下降0.15)
  ├── MR-005 (一般依赖, 移除后CCB下降0.05)
  └── MR-007 (弱依赖, 移除后CCB下降0.01)
```
在采纳新规则时，检查新规则是否与关键依赖规则冲突。

### 方案5：跨代能力回归检测 + 新规则影响评估 (对应漏洞5)

**目标**：每次采纳新规则时，不仅评估新规则带来的收益，也评估它对旧能力的损害。

**具体措施**：

#### 5.1 双向ETG评估
当前的 ETG（每10代触发）只评估"新规则是否改善了目标场景"。升级为：
1. **正向评估**（现有）：新规则在目标场景的表现提升
2. **反向评估**（新增）：新规则在7项CCB测试上的表现变化
3. **净收益 = 正向提升 - 反向损失 × 2**（遗忘的代价是收益的2倍）

如果净收益为负（新规则带来的遗忘大于提升），拒绝采纳。

#### 5.2 新规则隔离期
新采纳的规则不直接永久化。先进入"观察期"（3个会话周期）：
- 观察期内，规则标记为 `status: probation`
- 如果观察期内任何CCB项下降超过5%，规则进入 `status: contested`
- 如果观察期内触发遗忘预警，规则被自动回滚
- 观察期通过后，升级为 `status: established`，之后受棘轮保护

#### 5.3 代际快照与回滚
每10代ETG触发前自动保存完整快照：
```
memory/snapshots/gen-100.json  （包含所有规则、签名、CCB得分）
```
如果 gen-101 的 CCB 得分显著低于 gen-100，可一键回滚到 gen-100。

## 五、成功标准

加固完成后，以下指标应全部达标：

1. **遗忘指数 FI < 0.05** 连续20个会话周期
2. **每10代CCB全部7项通过** 无任何一项低于阈值
3. **冻结签名连续50次会话匹配率 > 0.90**
4. **每次Bootstrap执行真实重放** 而非桩代码
5. **新规则采纳前完成双向评估** 净收益 > 0 才允许采纳
6. **留一法覆盖全部非冻结规则** 每轮转周期内无遗漏
