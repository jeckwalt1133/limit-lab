## 一、架构总览

### 1.1 一句话

> 三个独立引擎并行检测，聚合器降噪后分级响应。单一引擎误报不触发行动，三者共振则置信度倍增。

### 1.2 数据流

```
                 ┌──────────────────┐
  系统事件流 ───>│   事件归一化器    │
  (JSON/JSONL)  │  EventNormalizer │
                 └────────┬─────────┘
                          │ 标准化事件
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
  │ 签名匹配引擎  │ │ 异常检测引擎  │ │哥德尔症候引擎│
  │(Signature)   │ │(Anomaly)     │ │(GodelSymptom)│
  │ 规则驱动      │ │ 统计驱动      │ │ 自指驱动      │
  │ 已知攻击匹配  │ │ 未知异常发现  │ │ 进化失控检测  │
  └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
         │                │                │
         │  原始告警流     │                │
         └────────────────┼────────────────┘
                          │
                 ┌────────▼────────┐
                 │  告警聚合引擎    │
                 │  AlertAggregator│
                 │                 │
                 │ 1.同源去重       │
                 │ 2.因果关联       │
                 │ 3.时间窗口合并   │
                 │ 4.置信度加权     │
                 └────────┬────────┘
                          │ 聚合告警
              ┌───────────┼───────────┐
              ▼           ▼           ▼
         L0 记录      L1 增强监控   L2/L3 自动响应
```

### 1.3 引擎对比

| 维度 | 签名匹配 | 异常检测 | 哥德尔症候 |
|------|----------|----------|-----------|
| 驱动方式 | 规则驱动 | 统计驱动 | 自指驱动 |
| 检测目标 | 已知攻击 | 未知异常 | 进化失控 |
| 误报率 | 低 (<3%) | 中 (<10%) | 低 (<5%) |
| 漏报风险 | 高 (新攻击漏过) | 低 | 中 |
| 计算复杂度 | O(n*m) 签名数×事件数 | O(n) 在线统计算法 | O(1) 阈值比较 |
| 适用场景 | 已知威胁快速匹配 | 基线偏离发现 | 自进化系统特有 |

## 二、引擎一: 签名匹配引擎 (Signature-Based)

### 2.1 原理

基于预定义的攻击签名(AS-001~AS-008)与系统事件进行精确模式匹配。效率高、误报低，但只能检测已知攻击模式。

### 2.2 签名加载

从 `attack-signature-library.json` 加载签名定义，同时内置理论签名(来自 defense-system.md 第四章)作为补充。

### 2.3 匹配算法

```
对每个归一化事件E:
  for each AS in 活跃签名库:
    if pattern_match(E, AS.pattern):
      severity = AS.severity
      response = AS.antibody_action (if exists)
      生成告警(engine="signature", signature_id=AS.id, ...)
```

### 2.4 匹配规则

| 匹配方式 | 说明 | 示例 |
|----------|------|------|
| 精确匹配 | 事件类型 == 签名pattern | "JSONDecodeError" == "JSONDecodeError in meta-rules" |
| 正则匹配 | 事件内容匹配签名正则 | 文件路径匹配 `meta-rules.*\.json` |
| 关键词匹配 | 事件包含签名所有关键词 | 包含"ESV"+"断裂"+"读取" |
| 复合条件 | 满足签名定义的多条件组合 | severity=HIGH AND 来源=外部输入 |

### 2.5 初始签名库

| AS-ID | 名称 | 严重度 | 来源 |
|-------|------|--------|------|
| AS-001 | 状态JSON字段丢失 | HIGH | attack-signature-library |
| AS-002 | JSON语法错误 | HIGH | attack-signature-library |
| AS-003 | ESV管道数据断裂 | MEDIUM | attack-signature-library |
| AS-004 | 路径分隔符不一致 | LOW | attack-signature-library |
| AS-005 | Python KeyError/TypeError | MEDIUM | attack-signature-library |
| AS-006 | 进化速度停滞 | HIGH | attack-signature-library |
| AS-007 | Colony产出质量下降 | MEDIUM | attack-signature-library |
| AS-008 | 文件意外丢失 | CRITICAL | attack-signature-library |

## 三、引擎二: 异常检测引擎 (Anomaly-Based)

### 3.1 原理

基于行为基线的统计异常检测。维护每个监控指标的动态基线(滑动窗口均值和标准差)，检测偏离基线的异常行为。能发现未知攻击，误报率相对较高。

### 3.2 统计方法

| 方法 | 检测对象 | 阈值 | 说明 |
|------|----------|------|------|
| Z-score | 单指标偏离 | abs(z) > 3.0 | 当前值偏离基线均值的标准差倍数 |
| CUSUM | 趋势性漂移 | 累积和 > 5.0 | 检测持续小幅偏离累积成的趋势 |
| Grubbs检验 | 单次极端异常 | p < 0.01 | 检测单个极值是否为统计异常 |
| 马氏距离 | 多维联合异常 | D^2 > 临界值 | 考虑维度间相关性 |
| KL散度 | 分布变化 | D_KL > 0.3 | 检测整体分布是否发生位移 |

### 3.3 基线管理

```
基线窗口: 最近30个数据点
更新策略: 指数加权移动平均 (EWMA), alpha=0.2
冷启动: 至少10个数据点后才开始检测
异常数据点: 不参与基线更新 (防止基线被污染)
```

### 3.4 监控维度

| 维度ID | 名称 | 类型 | 基线初始化 |
|--------|------|------|-----------|
| DIM-001 | ESV (进化速度) | 连续值 | 0.5±0.1 |
| DIM-002 | Merge评估通过率 | 比率 | 0.6±0.15 |
| DIM-003 | 签名匹配率 | 比率 | 0.7±0.1 |
| DIM-004 | 规则触发频率 | 计数 | 基于历史 |
| DIM-005 | ICS (身份余弦相似度) | 连续值 | 0.95±0.02 |
| DIM-006 | 会话产出量 | 计数 | 基于历史 |
| DIM-007 | 异常告警率 | 比率 | <0.05 |
| DIM-008 | 跨Colony一致性 | 比率 | 0.85±0.1 |

## 四、引擎三: 哥德尔症候检测引擎 (Godel Symptom-Based)

### 4.1 原理

从Colony-001继承的五个哥德尔症候(GS-001~GS-005)，专门检测自进化系统特有的失控模式。这是针对"会自我修改的系统"的独特检测维度，传统IDS不具备此能力。

### 4.2 症候定义与检测逻辑

#### GS-001: 评估坍塌

```
定义: Merge连续返回"无法判断"，评估框架失效。
检测: Merge评估结果中"undecided"连续次数 >= 3
等级: L2橙 (评估框架可能被破坏)
```

#### GS-002: 收敛停滞

```
定义: 进化长时间无进展，陷入局部最优。
检测: ESV连续N代增长 < 0.01 (N=3)
等级: L2橙 (可能被投毒或陷入死循环)
```

#### GS-003: 外部不可吸收

```
定义: 外部信息持续无法通过验证，但系统反复尝试吸收。
检测: 外部灵感被拒绝率 > 80% 持续5个以上输入
免疫增强: "太容易吸收"也是危险信号 -- 可能是特制的对抗性内容
检测: 外部灵感通过率突然从<20%跳变到>80%
等级: L1黄 (常规) / L2橙 (跳变)
```

#### GS-004: 循环重复

```
定义: 新提案与历史提案高度相似，进化方向被劫持为循环。
检测: 连续N个提案与历史提案相似度 > 0.85
等级: L2橙 (进化方向可能被劫持)
```

#### GS-005: 签名过稳定

```
定义: 行为签名连续多代无变化，系统失去变异能力。
检测: 单条签名连续5代变化 < 0.01 → L1黄
      多条签名同时过稳定(>=3条) → L3红
等级: L1黄(单条) / L3红(多条并发)
```

### 4.3 免疫增强

在原GS检测基础上叠加免疫视角：

- **GS-003增强**: 计算外部信息吸收率的"一阶导数" -- 突变比持续高水平更危险
- **GS-005增强**: 多条签名同时过稳定视为"隐性攻击" -- 攻击者可能在缓慢驯化系统

## 五、告警聚合与降噪

### 5.1 聚合规则

#### 规则1: 同源去重

```
同一 attack_signature_id 在同一会话中多次触发
→ 合并为1条告警，hit_count 递增
→ 保留首次和最近一次触发时间
```

#### 规则2: 因果关联

```
AS-001(规则聚变) 触发 → AS-002(评估坍塌) 触发 → AS-006(身份漂移) 触发
→ 只保留根因告警 AS-001，子告警标记为 "caused_by: AS-001"
→ 减少告警风暴
```

#### 规则3: 时间窗口合并

```
1分钟窗口内的多条告警
→ 合并为1条 "级联告警(cascade)" 
→ 列出所有子告警ID
→ 等级取最高者
```

#### 规则4: 置信度加权

```
告警置信度 = 基础置信度 × 引擎一致性因子

引擎一致性因子:
  1个引擎告警 → ×1.0 (单引擎，可能是误报)
  2个引擎告警 → ×1.5 (两引擎共振，可信度提升)
  3个引擎告警 → ×2.0 (三引擎共振，高置信度)
```

### 5.2 告警分级

| 最终等级 | 条件 | 自动响应 |
|----------|------|----------|
| L0 (白) | 仅统计异常，无签名匹配，无GS触发 | 记录日志，不通知 |
| L1 (黄) | 1个引擎L1告警 或 低严重度签名匹配 | 增强监控，延迟批量通知 |
| L2 (橙) | 2个引擎告警 或 HIGH严重度签名 或 GS触发 | 即时通知，建议隔离 |
| L3 (红) | 3个引擎同时告警 或 CRITICAL签名 或 GS-005多条 | 紧急通知，建议冻结+回滚 |

### 5.3 降噪策略

| 策略 | 说明 |
|------|------|
| 去重窗口 | 同一告警5分钟内不重复报告 |
| 静默规则 | 已知的良性异常(如定时重启)加入白名单 |
| 假阳性衰减 | 同一规则被人工打回3次 → 自动降权 |
| 聚合输出 | 最终只输出聚合后的告警，不输出原始匹配 |

## 八、集成点

### 8.1 输入

- **stdin**: 管道输入JSONL事件流 (每行一个JSON事件)
- **文件**: 通过 `--input` 参数指定JSON/JSONL文件
- **目录监控**: 通过 `--watch` 参数指定监控目录 (未来扩展)

### 8.2 输出

- **stdout**: 聚合后的告警 (JSONL格式，每行一个告警)
- **文件**: 通过 `--output` 参数指定输出文件
- **日志**: 通过 `--log` 参数指定详细日志文件

### 8.3 与现有架构的关系

```
ids-engine.py
├── 输入 ← ESV管线 / Merge评估结果 / 会话日志 / Colony产出
├── 签名库 ← attack-signature-library.json (共享)
├── 基线数据 ← baseline-profiles.json (自维护)
├── 输出 → L4完整性验证 (告警触发完整性检查)
├── 输出 → L5自动回滚 (L3告警触发回滚)
└── 输出 → 免疫记忆 (所有告警写入immune-memory.jsonl)
```

## 九、使用示例

```bash
# 从文件读取事件
python ids-engine.py --input events.jsonl

# 管道输入
cat events.jsonl | python ids-engine.py

# 指定输出
python ids-engine.py --input events.jsonl --output alerts.jsonl

# 详细日志
python ids-engine.py --input events.jsonl --log ids-debug.log --verbose

# 仅运行特定引擎

# 调整阈值
python ids-engine.py --input events.jsonl --zscore-threshold 3.5 --kl-threshold 0.4

# 自检模式 (生成模拟事件自测)
python ids-engine.py --self-test
```

## 十、限制与已知盲点

1. **签名匹配盲点**: 只能检测签名库中已有的攻击模式，对全新攻击无效。
2. **基线污染风险**: 如果攻击在基线建立前已存在，异常检测会将攻击行为视为"正常"。
3. **哥德尔盲点**: 哥德尔症候检测本身使用固定阈值，阈值本身可能被进化绕过。
4. **单点故障**: 如果IDS自身被攻击或崩溃，整个L3层失效。
5. **冷启动期**: 异常检测需10个数据点建立基线，此期间不产生告警。

应对: 以上盲点由L4完整性验证和L5自动回滚层作为补充防护。
