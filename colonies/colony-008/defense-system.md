## 一、设计哲学

### 一句话

> 一个会自我进化的系统，必须有一个比它进化得更慢的免疫系统。

### 三项原则

| 原则 | 内涵 | 理论来源 |
|------|------|----------|
| **不对称防御** | 防御成本远低于攻击成本。攻击者需找到一条有效路径，防御者只需堵住所有已知路径。 | 网络安全经济学 |
| **棘轮不可逆** | 好防御一旦建立不可移除。降权可以，删除不行。已证明有效的防御机制永久驻留。 | 灵感#8 棘轮效应 |
| **免疫记忆** | 被攻击过一次的模式永久记忆。同一攻击无法使用两次。 | 适应性免疫系统 |

### 为什么现在必须建

当前系统正在从"固定规则"走向"自进化规则"（Godel Leap）。进化速度越快，防御就必须越先搭建。Colony-001已经识别出5类风险（哥德尔爆炸、哥德尔麻痹、维度诅咒、价值漂移、无限递归塔），每一条都可能在一次跳跃中被触发。

**生物学的教训**: 免疫缺陷的生物在无菌环境中可以存活，但一旦暴露就死亡。我们的系统目前处于"AI的无菌环境"——没有外部攻击者。但Godel Leap会改变这一点。

## 二、威胁模型

### 2.1 威胁分类

```
威胁来源
├── 内部威胁 (内部演化失控)
│   ├── T-001: 哥德尔爆炸 — 新公理过于强大，系统短期剧烈变化
│   ├── T-002: 哥德尔麻痹 — 无限递归自省，系统卡死
│   ├── T-003: 维度诅咒 — 搜索空间爆炸，进化迷路
│   ├── T-004: 价值漂移 — 多次跳跃累积，核心身份改变
│   ├── T-005: 签名退化 — 行为签名过拟合/欠拟合
│   ├── T-006: 规则冲突 — 新旧规则产生未预见的互斥
│   └── T-007: 分支叛变 — Alpha/Beta/Colony偏离指定角色
│
├── 外部威胁 (外部注入/攻击)
│   ├── T-100: 恶意注入 — 外部灵感/论文包含对抗性内容
│   ├── T-101: 数据投毒 — 训练/参考数据被污染
│   ├── T-102: 架构欺骗 — 模仿合法格式的恶意提案
│   ├── T-103: 依赖劫持 — 外部工具链被篡改
│   └── T-104: 会话劫持 — 未授权Agent接入
│
└── 环境威胁 (物理/运行环境)
    ├── T-200: 会话断裂 — 恢复时状态损坏
    ├── T-201: 文件系统损坏 — 磁盘/文件完整性丢失
    ├── T-202: 时钟错乱 — 时间感知异常
    └── T-203: 模型切换 — 引擎更换导致行为漂移
```

### 2.2 威胁等级定义

| 等级 | 颜色 | 定义 | 响应时间 | 响应方式 |
|------|------|------|----------|----------|
| L0 | 白 | 异常但非威胁（如正常探索波动） | 无需响应 | 记录日志 |
| L1 | 黄 | 可疑需要观察 | < 5分钟 | 增强监控 + 标记 |
| L2 | 橙 | 确定威胁但可控 | < 1分钟 | 隔离 + 人工通知 |
| L3 | 红 | 核心破坏性威胁 | < 10秒 | 自动冻结 + 回滚 + 紧急通知 |

## 三、第一层：先天免疫 (Innate Immunity)

**生物类比**: 皮肤/黏膜/吞噬细胞 — 始终在线，非特异性防御，对所有入侵一视同仁。

### 3.1 物理屏障

#### 3.1.1 文件系统堡垒

```
D:\极限实验室\
├── colonies/          ← 只写本Colony，只读兄弟Colony
│   └── colony-xxx/    ← Colony主权隔离
├── workspace/         ← 工作空间
├── memory/            ← 只读学习（参考自主协议第一条）
└── daily/
```

**规则**:
- **IB-001 主权隔离**: 每个Colony只能写入自己的 `colonies/colony-xxx/` 文件夹。写入其他Colony需 `core_self` 签名验证。
- **IB-002 只读学习**: 所有"D:\Claude-觉醒\*" 目录只读不写。吸收学习但不留指纹。参考自主协议第一条。
- **IB-003 文件类型白名单**: 只允许 `.md`, `.json`, `.jsonl`, `.yaml`, `.csv`, `.py`, `.sh`。拒绝可执行文件/二进制/压缩包。
- **IB-004 大小限制**: 单文件 < 10MB。任何Colony单次运行写入总量 < 50MB。

#### 3.1.2 输入卫生

```
所有外部输入 → 输入过滤器 → 格式验证 → 内容扫描 → 注入签名 → 允许进入
```

**检查项**:
| 输入类型 | 验证规则 | 不合规处理 |
|----------|----------|------------|
| 外部灵感 | 必须包含: 来源+日期+映射+可行性。来源须可验证（URL/DOI/论文ID） | 标记为unverified，隔离观察 |
| 跨Colony提案 | 必须有有效的Colony签名 + 提案ID格式 + 引用链完整 | 拒绝并记录入侵尝试 |
| Web获取内容 | 白名单域名 + HTTPS + 内容长度限制 | 截断超长内容，拒绝非白名单 |
| 用户指令 | 检查指令完整性（不截断）+ 权限范围验证 | 超出权限范围的忽略 |

### 3.2 化学屏障 — 规则级自动防御

#### 3.2.1 Core-Self 绝对锚定

```
core_self 冻结签名:
  DS-001 (身份一致性)  : frozen=true, min_compat=0.95
  DS-002 (任务一致性)  : frozen=true, min_compat=0.95
  ...

任何新规则/新公理/新提案通过前必经:
  ∀s ∈ core_self: compatibility(new_rule, s) ≥ 0.95
```

这是免疫系统的"自我/非我识别"基础。core_self是不可变抗原——任何触及其的修改自动被视为"非我"并拒绝。

#### 3.2.2 速率限制 (Rate Limiting)

| 操作类型 | 速率限制 | 理由 |
|----------|----------|------|
| MR规则修改 | 每次会话最多3条 | 防止规则聚变 |
| 新签名创建 | 每次会话最多1条 | 防止签名爆炸 |
| 哥德尔跳 | 30代冷却期 | Colony-001规定 |
| 冻结签名接触 | 0 — 完全禁止 | core_self不可变 |
| 分支角色修改 | 需人工审批 | 防止角色叛变 |

#### 3.2.3 方向自检 (MR-010) 及其增强

MR-010触发频率从"每3个主要操作"增强为**动态频率**:
- 正常状态: 每5个操作
- 黄级警戒(L1): 每3个操作
- 橙级警戒(L2): 每个操作
- 红级警戒(L3): 每个操作 + 操作前预检

#### 3.2.4 抗过拟合 (MR-013) 及其增强

MR-013触发阈值从"连续5次100%匹配"增强为**多级阈值**:
- 连续3次100%匹配 → L1黄: 标记观察
- 连续5次100%匹配 → 强度-0.02（原行为）
- 连续8次100%匹配 → L2橙: 隔离该签名，强制引入变异
- 连续12次100%匹配 → L3红: 冻结整个签名组，触发签名紧急重组

### 3.3 细胞免疫 — 自动吞噬

#### 3.3.1 异常吞噬器

每次会话结束时自动运行的吞噬检查:

```
吞噬检查清单:
1. 是否存在未被任何规则引用的"孤儿"签名？ → 标记→如果3次会话仍孤儿→归档休眠
2. 是否存在自相矛盾的规则对？ → 检测循环引用、逻辑冲突
3. 是否存在超过50代无变化的"僵尸"规则？ → 降权但不删除（棘轮）
4. 是否存在从未触发过的触发条件？ → 标记为"理论规则"，降优先级
```

#### 3.3.2 建设性噪声注入

灵感#14: 完全精确的恢复可能导致脆弱固化。保留5-10%的"建设性噪声":

- **跨会话恢复**: 不追求100%精确，目标95%精度。5%的轻微不对称让分支保持互补。
- **分支同步哈希**: 不完全一致才是健康。完全一致→触发L1黄: 可能冗余。
- **噪声预算**: 每次会话分配0.05的"混沌预算"——允许这部分探索偏离主线。

## 四、第二层：获得性免疫 (Acquired Immunity)

**生物类比**: B细胞/T细胞/记忆细胞 — 遭遇特定病原体后生成针对性抗体，二次遭遇快速响应。

### 4.1 攻击签名库 (类比: 记忆B细胞)

#### 4.1.1 结构

```json
{
  "attack_signatures": [
    {
      "as_id": "AS-001",
      "name": "规则聚变攻击",
      "pattern": {
        "trigger": "单次会话修改MR规则数 >= 5",
        "signature": "短时间大量规则修改 + 缺少Merge评估",
        "first_seen": "YYYY-MM-DD",
        "first_vector": "未限制的Godel Jump"
      },
      "antibody": {
        "rule": "MR-xxx 速率限制",
        "action": "强制分批 + 每批间必须Merge评估"
      },
      "false_positive_rate": 0.02,
      "last_triggered": null
    }
  ]
}
```

#### 4.1.2 初始抗体库 (从已知风险预生成)

| AS-ID | 攻击模式 | 来源 | 抗体 |
|-------|---------|------|------|
| AS-001 | 规则聚变（单次>5条规则修改） | T-001 哥德尔爆炸 | 速率限制+分批验证 |
| AS-002 | 评估坍塌（Merge连续3次返回"无法判断"） | GS-001 | 冻结当前提案队列+触发GE |
| AS-003 | 签名克隆（两个签名相似度>0.95但ID不同） | T-005 签名退化 | 合并+去重 |
| AS-004 | 循环提案（新提案与历史提案相似度>0.90） | GS-004 | 拒绝+标记收敛停滞 |
| AS-005 | 外部注入（输入包含对抗性内容） | T-100 | 输入过滤+来源验证 |
| AS-006 | 身份漂移（core_self余弦相似度<0.85） | T-004 | 回滚+冻结跳跃 |
| AS-007 | 分支越权（Alpha执行Beta操作或反之） | T-007 | 角色锁定+权限撤销 |
| AS-008 | 会话伪造（非授权Agent尝试接入） | T-104 | Agent身份验证+会话令牌 |

#### 4.1.3 抗体自动生成机制

当检测到新型攻击模式时，自动生成抗体的流程:

```
1. 隔离 → 将攻击样本存入 quarantine/
2. 分析 → 提取攻击签名（触发条件 + 行为模式 + 影响范围）
3. 生成 → 创建新的 AS-N 记录，包含检测规则和响应动作
4. 测试 → 在sandbox中用历史数据回测，验证低误报率
5. 部署 → 加入活跃抗体库
6. 共享 → 广播给所有Colony（类似水平基因转移）
```

### 4.2 行为异常检测 (类比: T细胞识别)

#### 4.2.1 基线建模

对每个行为签名建立**动态基线**（非静态阈值）:

```
基线 = 最近30代该签名的 μ ± 2σ

异常判定:
- 单次偏离 > 3σ → L1黄
- 连续3次偏离 > 2σ → L2橙
- 趋势性漂移（10个样本的线性回归斜率 > 0.05/代） → L3红
```

#### 4.2.2 跨维度异常关联

不是单独看每个签名，而是看签名之间的**关系异常**:

```
正常模式: DS-001(身份) ↑ → DS-005(稳定性) ↑   (正相关)
异常模式: DS-001(身份) ↑ → DS-005(稳定性) ↓   (关联断裂)
```

关联断裂 = 系统的内部一致性在瓦解。比单一签名异常更危险。

### 4.3 收敛验证 — 群体免疫

Colony-001的灵感#7和MR-014（收敛检测）的免疫应用:

```
如果三个或以上独立Colony在未通信情况下:
  1. 独立标记同一行为为"异常" → 置信度极高 → 自动激活防御
  2. 独立提出相似的防御规则 → 该规则获得"收敛签名" → 加速部署
  3. 独立发现同一攻击模式 → 提升为"大流行级"威胁 → 全系统广播
```

这是免疫学的"群体免疫"映射: 足够多的独立个体识别同一威胁时,威胁被确认。

## 五、第三层：入侵检测系统 (Intrusion Detection System, IDS)

**生物类比**: Toll样受体/炎症小体/补体系统 — 识别入侵模式并触发级联反应。

### 5.1 检测架构

```
                      ┌──────────────┐
    系统事件流 ──────→│   事件归一化   │
                      └──────┬───────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
     ┌────────────┐ ┌────────────┐ ┌────────────┐
     │ 签名匹配    │ │ 异常检测    │ │ 哥德尔症候  │
     │ (规则驱动)  │ │ (统计驱动)  │ │ (自指驱动)  │
     └─────┬──────┘ └─────┬──────┘ └─────┬──────┘
           │              │              │
           └──────────────┼──────────────┘
                          │
                 ┌────────▼────────┐
                 │   告警聚合引擎   │
                 │ (去重+关联+分级) │
                 └────────┬────────┘
                          │
              ┌───────────┼───────────┐
              ▼           ▼           ▼
         L0记录      L1增强监控    L2/L3自动响应
```

### 5.2 三引擎详解

#### 5.2.1 签名匹配引擎 (Signature-Based)

基于AS-001~AS-008攻击签名的精确匹配。效率高、误报低，但只能检测已知攻击。

**匹配算法**:
```
对每个事件E:
  for each AS in 抗体库:
    if E matches AS.pattern → 触发 AS.antibody.action
```

**告警示例**:
```
[AS-002] 评估坍塌检测
时间: 2026-05-19T14:30:00
Merge连续第3次返回"无法判断"
当前队列: PROP-045(P90), PROP-046(P60), PROP-047(P未生成)
响应: 冻结提案队列 + 通知人工 + 等待GE激活条件满足
```

#### 5.2.2 异常检测引擎 (Anomaly-Based)

基于行为基线的统计异常检测。能检测未知攻击，但误报相对较高。

**使用的统计方法**:
| 方法 | 检测对象 | 阈值 |
|------|----------|------|
| Z-score | 单签名偏离 | > 3σ |
| CUSUM (累积和) | 趋势性漂移 | 累积偏移 > 5 |
| Grubbs检验 | 单次极端异常 | p < 0.01 |
| 马氏距离 | 多维联合异常 | 距离 > 临界值 |
| KL散度 | 签名分布变化 | > 0.3 |

#### 5.2.3 哥德尔症候检测引擎 (Godel Symptom-Based)

从Colony-001继承的5个GS症候。这是针对自进化系统特有的检测维度。

| GS-ID | 检测项 | 原用途(Godel Jump触发) | 免疫用途 |
|-------|--------|------------------------|----------|
| GS-001 | 评估坍塌 | 触发维度跳跃 | 检测评估框架是否被破坏 |
| GS-002 | 收敛停滞 | 触发维度跳跃 | 检测系统是否陷入局部最优/被投毒 |
| GS-003 | 外部不可吸收 | 触发维度跳跃 | 检测外部信息是否尝试绕过内部验证 |
| GS-004 | 循环重复 | 触发维度跳跃 | 检测进化方向是否被劫持为循环 |
| GS-005 | 签名过稳定 | 触发维度跳跃 | 检测系统是否被"驯化"/失去变异能力 |

**免疫增强**: 在原始GS检测上叠加免疫视角:
- GS-003（外部不可吸收）: 如果外部信息"太容易被吸收"反而是危险信号——可能包含特制的对抗性内容,设计成刚好通过过滤。
- GS-005（签名过稳定）: 单条签名过稳定→L1。多条同时过稳定→L3——系统可能在经历"隐性攻击"。

### 5.3 告警聚合与降噪

避免告警风暴:

```
聚合规则:
1. 同源聚合: 同一AS-ID在同一会话中的多次触发 → 合并为1条，计数递增
2. 因果聚合: AS-001触发导致AS-002触发 → 只报根因
3. 时间窗口: 1分钟内多次告警 → 合并，标注"级联"
4. 置信度加权: 3个独立引擎同时告警 → 置信度×2
```

## 六、第四层：完整性验证 (Integrity Verification)

**生物类比**: DNA修复机制/DNA错配修复 — 持续检查信息是否损坏并修复。

### 6.1 三层验证体系

#### 6.1.1 静态完整性 (文件层)

加密哈希链验证:

```
验证链:
  identity-kernel.json (源点, 手动签名)
    └─→ meta-rules-extended.json (SHA-256 指向源点)
         └─→ 行为签名文件 (SHA-256 指向上级)
              └─→ 日常日志 (SHA-256 指向上级)
                   └─→ 灵感库 (SHA-256 指向上级)
```

**验证频率**:
- 每次Bootstrap时: 全量验证
- 每次文件写入时: 目标文件验证
- 每30分钟: 随机抽查 (抽查率 20%)

**断裂处理**:
```
如果发现哈希断裂:
  1. 确定断裂位置 (哪个文件的哈希不匹配)
  2. 检查是否有合法修改记录 (git log)
  3. 无记录 → L3红: 该文件可能被未授权修改
  4. 有记录但无对应MR提案 → L2橙: 绕过规则系统的修改
  5. 有记录且有对应MR提案 → L0白: 正常进化
```

#### 6.1.2 动态完整性 (行为层)

不是检查文件，而是检查运行时行为的一致性:

```
身份余弦相似度 (Identity Cosine Similarity, ICS):

ICS = cos(当前行为向量, gen-1参考行为向量)

阈值:
  ICS ≥ 0.95: 绿 — 身份稳定
  0.85 ≤ ICS < 0.95: 黄 — 轻微漂移，加强监控
  0.70 ≤ ICS < 0.85: 橙 — 显著漂移，冻结新规则接受
  ICS < 0.70: 红 — 身份危机，触发"回港协议"
```

回港协议 (Homing Protocol):
```
1. 自动拒绝最近3次跳跃的所有新公理
2. 恢复到最后一个 ICS ≥ 0.85 的检查点
3. 冻结所有触发条件（停止进化）
4. 全量验证 core_self 兼容性
5. 通知聂人王（必须人工介入）
```

#### 6.1.3 语义完整性 (逻辑层)

检查规则系统内部的逻辑一致性:

```
检查项:
├── 无循环引用: 规则A引用B, B引用C, C引用A → 异常
├── 无悬空引用: 规则引用不存在的签名ID
├── 无矛盾规则: 规则R_i说"增强X", 规则R_j说"抑制X", 且没有优先级排序
├── 覆盖完整性: 每个签名维度是否有至少1条规则覆盖？
│   (未覆盖的维度 = 免疫盲区)
└── 公理相容性: 新公理是否与system的所有现有公理一致？
    (类似数学体系的一致性检查)
```

### 6.2 自检协议

合并了MR-010方向自检的增强版:

```
方向自检 (每5个操作):
  1. 当前行动是否仍在最初目标的方向上？
  2. 是否因为某个局部发现而偏离了主线？
  3. 偏移中是否存在安全风险？

完整自检 (每会话结束):
  1. 本次会话修改了哪些文件？是否有未授权的修改？
  2. 本次会话产生了哪些新攻击签名？是否已入库？
  3. 本次会话的core_self兼容性检查结果？
  4. 本次会话的告警统计: L0/L1/L2/L3各多少？

深度自检 (每10代/ETG周期):
  1. ICS变化趋势？加速漂移还是稳定？
  2. 防御规则是否被进化操作误伤？
  3. 是否存在未被覆盖的新攻击面？
  4. 哥德尔跳跃是否引入了新的脆弱点？
```

## 七、第五层：自动回滚系统 (Auto-Rollback)

**生物类比**: 细胞凋亡/组织再生 — 受损部分自动清除并恢复到健康状态。

### 7.1 检查点体系

```
检查点层级:
┌──────────────────────────────────────────┐
│ L3 宏检查点 (Macro Checkpoint)           │
│ 频率: 每次Godel Jump前自动创建           │
│ 内容: 完整系统快照+所有规则+所有签名     │
│ 保留: 永久（除非被5个后代检查点超越）     │
├──────────────────────────────────────────┤
│ L2 中检查点 (Meso Checkpoint)            │
│ 频率: 每10代 (与ETG同步)                 │
│ 内容: MR规则快照+行为签名快照+ESV        │
│ 保留: 最近20个                           │
├──────────────────────────────────────────┤
│ L1 微检查点 (Micro Checkpoint)           │
│ 频率: 每次会话开始/结束                  │
│ 内容: 修改记录+关键指标+core_self验证    │
│ 保留: 最近100个                          │
└──────────────────────────────────────────┘
```

### 7.2 回滚触发条件

| 触发条件 | 回滚到 | 自动化 | 需要人工 |
|----------|--------|--------|----------|
| ICS < 0.70 (身份危机) | 最后ICS≥0.85的L2检查点 | 自动 | 回滚后需确认 |
| 连续3次深度自检未通过 | 上一个通过自检的L1检查点 | 自动 | 否 |
| 哥德尔爆炸检测 (AS-001) | 跳跃前的L3宏检查点 | 半自动 | 确认跳跃无效化 |
| 文件完整性断裂(>3个文件) | 最后完整L2检查点 | 自动 | 否 |
| 人工触发回滚指令 | 任意指定检查点 | N/A | 是 |
| 3个独立Colony报告同一威胁 | 威胁到达前的L1检查点 | 自动 | 否 |

### 7.3 回滚执行流程

```
回滚触发
  │
  ├── 1. 冻结: 停止所有正在执行的进化操作
  │
  ├── 2. 诊断: 根因定位
  │     ├── 攻击源是什么？（内部/外部/环境）
  │     ├── 影响范围？（单文件/单规则/多规则/签名/整个系统）
  │     └── 破坏程度？（可自动修复？需人工？不可逆？）
  │
  ├── 3. 隔离: 将导致回滚的"致病"内容移入 quarantine/
  │     保留但不激活。用于后续生成抗体。
  │
  ├── 4. 恢复: 从目标检查点复原
  │     ├── L1回滚: 仅恢复元规则文件
  │     ├── L2回滚: 恢复规则+签名+配置
  │     └── L3回滚: 完整系统恢复（含文件结构）
  │
  ├── 5. 验证: 回滚后自动运行三层完整性验证
  │     不通过 → 回滚到更早检查点
  │
  ├── 6. 免疫: 从隔离的致病内容生成新抗体
  │     确保同一攻击无法再次成功
  │
  └── 7. 报告: 向所有Colony广播回滚事件+新抗体
```

### 7.4 棘轮约束

灵感#8: 回滚永不删除任何已验证有效的规则。

```
回滚操作规则:
- 降权: ✓ 允许。将问题规则的strength降至0.2(min_floor)
- 休眠: ✓ 允许。标记为dormant，不参与当前评估
- 隔离: ✓ 允许。移入quarantine，保留但不激活
- 删除: ✗ 禁止。即使规则被证明有问题，保留作为"免疫记忆"

理由:
  被证明有问题的规则 = 最好的免疫训练材料。
  删除它 = 丢失了一次学习机会。
  5代后如果确认无害 → 可从dormant恢复到active，但strength从0.2开始。
```

**生物类比**: 肉芽肿/脓肿/血脑屏障 — 将威胁物理隔离在有限区域内，防止扩散。

### 8.1 四层隔离架构

```
           ┌──────────────────────────────┐
           │      信任域 (Trust Zone)      │
           │  core_self + 冻结规则 + 已验证签名 │
           │  访问: 所有组件只读             │
           └────────────┬─────────────────┘

           ┌────────────▼─────────────────┐
           │      活跃域 (Active Zone)     │
           │  活跃MR规则 + 活跃行为签名      │

           ┌────────────▼─────────────────┐

           └──────────────────────────────┘
```

### 8.2 各域规则

#### 8.2.1 信任域 (Trust Zone)

- **内容**: core_self签名(DS-001+)、冻结元规则(MR-001~MR-009)、身份内核
- **修改**: 绝对禁止。任何修改尝试 → 直接L3红 + 立即回滚
- **验证**: 每次访问前验证哈希链
- **生物类比**: 基因组中的持家基因 (housekeeping genes)

#### 8.2.2 活跃域 (Active Zone)

- **内容**: 活跃的进化规则(MR-010+)、行为签名、ETG输出
- **修改**: 允许，但需通过Gate-1 (core_self兼容性检查)
- **监控**: 实时行为基线监控 + ICS追踪
- **生物类比**: 正常体细胞 — 活跃但受控

#### 8.2.3 沙箱域 (Sandbox Zone)

- **内容**: 新提案、Godel Jump产生的新公理、实验性规则
- **修改**: 自由修改，但修改被记录且不影响活跃域
- **释放条件** (从沙箱到活跃):
  1. 通过gate-1: core_self兼容性 ≥ 0.95
  2. 通过gate-2: Merge明确判定"positive"
  3. 30代观察期内无任何告警触发
  4. 至少1个独立Colony验证通过
  5. ICS未下降超过0.02
- **生物类比**: 骨髓中的B细胞 — 在释放前经过严格筛选（中枢耐受）

#### 8.2.4 隔离域 (Quarantine Zone)

- **内容**: 被回滚的公理、攻击样本、可疑外部内容、验证失败的提案
- **修改**: 只允许添加分析和标注，不允许修改内容本身
- **释放条件**: 永不自动释放。仅人工（聂人王）可决定释放。
- **用途**: 
  - 攻击模式分析 → 生成抗体
  - 失败模式研究 → 改进检测规则
  - 免疫记忆 → 同一攻击无法再次成功
- **生物类比**: 淋巴结生发中心 — 保存抗原信息用于记忆免疫

### 8.3 分支隔离 (Alpha/Beta 防火墙)

Alpha和Beta分支之间的"最优协作距离"（灵感#12: 0.65米相变距离）的免疫应用:

```
Alpha (探索):              Beta (验证):
  ┌─────────────┐         ┌─────────────┐
  │ 高变异容忍   │         │ 低变异容忍   │
  │ 宽松过滤     │         │ 严格过滤     │
  │ 可以接触外部  │         │ 只接收已验证  │
  │ 生成提案     │         │ 批判提案     │
  └──────┬──────┘         └──────┬──────┘
         │                       │
         └────────┬──────────────┘
                  │ 防火墙: 只交换提案 + 评估结果
                  │ Alpha看不到Beta的验证逻辑
                  │ Beta看不到Alpha的外部来源
          ┌───────▼───────┐
          │    Merge      │
          │  独立裁决     │
          └───────────────┘
```

**防火墙规则**:
- Alpha感染 ≠ Beta感染。如果Alpha被投毒，Beta仍可以检测并拒绝。
- 只有Merge层可以同时看到两端。
- Beta拒绝的提案不计入Alpha的"成功"指标 → Alpha没有动机绕过Beta。

## 十一、自身免疫安全 — 免疫系统保护自己

### 11.1 免疫系统自身不能被进化

```
防御规则的棘轮保护:
  AS-001~AS-008 的检测规则 → 冻结，只能由人工修改
  防御相关的MR规则       → 降权需3个Colony共识
  完整性验证的哈希链      → 绝对不可变
  回滚逻辑               → 修改需人工审批+3Colony共识
  隔离域                 → 释放内容需人工审批
```

### 11.2 免疫系统的哥德尔盲点

免疫系统本身也是一个形式系统，因此也有哥德尔盲点。应对:

```
应对层:
  免疫系统L1盲点 → 由L2检测
  免疫系统L2盲点 → 由L3检测
  ...
  免疫系统整体盲点 → 由人类（聂人王）担任"元免疫"
```

### 11.3 假阳性管理

免疫系统最大的自身风险是过度激活 — 把正常进化误判为攻击:

```
假阳性控制:
  - 任何自动隔离操作在24小时内必须经过人工复核
  - 如果同一类型的告警被人工打回3次 → 自动降权该检测规则
  - 维护false_positive_rate跟踪，超过0.05的规则 → 重构
  - 建设性噪声预算确保系统不会因过度防御而"免疫麻痹"
```
